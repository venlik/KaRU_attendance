1. System Overview

The KaRU Attendance Tracker is a web-based attendance management system developed for Karatina University. The system enables students to register for accounts, sign attendance for classes, and view their attendance records. Lecturers have administrative capabilities to manage student accounts, approve registrations, generate attendance reports, and oversee the entire attendance tracking process.

The system uses PHP for server-side processing, MySQL for database management, and incorporates responsive HTML/CSS for a user-friendly interface across all devices including desktops, tablets, and mobile phones.
Key Features

    Student self-registration with account verification workflow

    Secure login system with password encryption

    Role-based access control for students and lecturers

    Daily attendance signing for students

    Real-time attendance reporting for lecturers

    Bulk student account creation by lecturers

    Student account approval/rejection system

    Password change functionality with security requirements

    Responsive design for all device sizes

5. Module Documentation
5.1 Configuration Module (config.php)

Purpose: Establishes database connection and provides connection object to all other PHP files.

Functionality:

    Defines database credentials (host, username, password, database name)

    Creates MySQLi connection object

    Handles connection errors gracefully

Dependencies: Required at the top of every PHP file that needs database access.

Code Explanation:
php

$conn = new mysqli($servername, $username, $password, $dbname);

Creates a new MySQLi connection. If connection fails, the script terminates with an error message.
5.2 Landing Page (index.html)

Purpose: Serves as the entry point for the application, presenting users with navigation options to register or login.

Components:

    Header with system title and university name

    Navigation bar with links to Home, Register, and Login

    Authentication card with prominent buttons for registration and login

    Footer with copyright information

User Flow:

    New users click "Student Registration" to create an account

    Existing users click "Login to System" to access their dashboard

    Navigation links provide alternative access to registration and login pages

5.3 Registration Module (register.php)

Purpose: Allows new students to create an account in the system with pending approval status.

Validation Rules:

    Registration Number Format: Letter followed by three digits, slash, four digits, optional letter, slash, two digits (Example: S123/0001X/24)

    Email Format: Standard email format with @ and domain

    Phone Number: Kenyan format starting with 01 or 07 followed by 8 digits (10 digits total)

    Password Requirements:

        Minimum 8 characters

        At least one uppercase letter

        At least one lowercase letter

        At least one digit

        At least one special character

Workflow:

    User submits registration form

    Server validates input and checks for existing registration number or email

    If registration number or email already exists, appropriate error message is displayed

    New account is inserted with account_status = 'pending'

    User receives confirmation that verification is required before login

    User is redirected to login page

Security:

    Passwords are hashed using password_hash() before storage

    Input sanitization using mysqli_real_escape_string()

    JavaScript validation for immediate feedback

    Server-side validation as backup

5.4 Login Module (login.php)

Purpose: Authenticates users and redirects them to their appropriate dashboards based on role.

Login Flow:

    User enters registration number and password

    System queries the students table for the registration number

    Account status is checked before password verification

    Password verification using password_verify()

    Session variables are set upon successful login

    Users with must_change_password = 1 are redirected to change password page

    Lecturers go to lecturer-dashboard.php

    Students go to dashboard.php

Account Status Handling:

    pending: User sees message that account awaits verification

    rejected: User sees message that registration was rejected

    verified: User proceeds with login

Session Variables Set:

    user_id: Database ID of the user

    fullname: User's full name

    reg_number: Registration number or staff ID

    role: Either 'student' or 'lecturer'

5.5 Student Dashboard (dashboard.php)

Purpose: Displays student's attendance summary and provides navigation to attendance functions.

Access Control: Only users with role 'student' can access this page.

Displayed Information:

    Student's registration number

    Total number of classes attended (based on attendance records)

    Number of days present

    Attendance percentage

    Color-coded percentage (green for 75% and above, red for below 75%)

Navigation Options:

    Sign Today's Attendance button

    View Full Record button

    Change Password link

    Logout link

5.6 Mark Attendance Module (mark-attendance.php)

Purpose: Allows students to sign their attendance for the current day.

Workflow:

    System checks if student has already signed attendance for today

    If already signed, message displays that attendance has already been recorded

    If not signed, student clicks the large attendance button

    System creates a new attendance record with status 'Present' for today's date

    Confirmation message appears and page refreshes

Prevention of Multiple Submissions:

    Database query checks for existing record before insertion

    Students can only sign once per day

5.7 View Attendance Module (view-attendance.php)

Purpose: Displays complete attendance history for the logged-in student.

Display Format:

    Table showing date and status for each attendance record

    Records ordered from newest to oldest

    Status column color-coded (green for Present, red for Absent)

    Empty state message when no records exist

5.8 Change Password Module (change-password.php)

Purpose: Allows both students and lecturers to change their passwords securely.

Access Control: Requires user to be logged in (either role).

Validation Checks:

    Current password must match the stored password

    New password must be different from current password

    New password and confirm password must match

    New password must be at least 6 characters

Post-Change Actions:

    must_change_password flag is set to 0

    Success message displayed

    Automatic redirect to appropriate dashboard after 2 seconds

5.9 Lecturer Dashboard (lecturer-dashboard.php)

Purpose: Provides lecturers with an overview of attendance statistics and pending actions.

Access Control: Only users with role 'lecturer' can access.

Statistics Displayed:

    Total number of verified students

    Number of students who signed attendance today

    Number of students absent today

    Number of pending student approvals

    Today's overall attendance rate percentage

Notification System:

    Alert banner appears when pending approvals exist

    Pending count badge appears on navigation tab

Quick Actions:

    Review Now button for pending approvals

    View Full Today's Report button

5.10 Attendance Report Module (attendance-report.php)

Purpose: Generates a detailed report of today's attendance for all verified students.

Report Components:

    Summary cards showing total expected students, present count, absent count

    Detailed table with student registration number, name, course, and status

    Color-coded status column (green for Present, red for Absent)

Data Source:

    Left join between students table and attendance table

    Shows 'Absent' for students without an attendance record for today

5.11 All Students Module (all-students.php)

Purpose: Displays all student accounts with filtering capabilities.

Access Control: Lecturers only.

Filter Options:

    All students

    Verified students only

    Pending students only

    Rejected students only

Displayed Information:

    Registration number

    Full name

    Email address

    Phone number

    Course/Program

    Account status with color-coded badge

    Who created the account

    Registration date

5.12 Pending Approvals Module (pending-approvals.php)

Purpose: Allows lecturers to approve or reject student registration requests.

Access Control: Lecturers only.

Workflow:

    Displays all students with account_status = 'pending'

    Lecturer reviews each student's information

    Clicking Approve sets account_status to 'verified'

    Clicking Reject sets account_status to 'rejected'

    Confirmation dialog before each action

    Alert confirms the action and page refreshes

Visual Indicators:

    Warning banner showing count of pending approvals

    Empty state message when no pending approvals exist

5.13 Create Students Module (create-students.php)

Purpose: Enables lecturers to create multiple student accounts at once in bulk.

Access Control: Lecturers only.

Input Format:

    Course/Program for all students in the batch

    Default password (students will be forced to change on first login)

    CSV-formatted student list with columns: Reg Number, Full Name, Email, Phone

CSV Format Example:
text

X123/4567Y/24,John Mwangi,john@example.com,0712345678
Y987/6543P/24,Jane Wanjiku,jane@example.com,0723456789

Processing Logic:

    Each line is parsed using str_getcsv()

    Duplicate registrations or emails are skipped

    New accounts are created with account_status = 'verified'

    All bulk-created accounts have must_change_password = 1

    Summary of created and skipped accounts is displayed

5.14 Create Lecturer Module (create-lecturer.php)

Purpose: One-time setup script to create the initial lecturer account.

Important Security Note: This file should be deleted after successful execution.

Default Lecturer Credentials:

    Staff ID: LEC001

    Password: LecSecure@2024

    Name: Dr. Jane Muthoni

    Department: Computer Science

Post-Creation: Lecturer must change password on first login.

Purpose: Terminates user session and redirects to homepage.

Actions Performed:

9. Installation Guide
Step 1: Set Up Local Server Environment

If using XAMPP:

    Download and install XAMPP from https://www.apachefriends.org/

    Launch XAMPP Control Panel

    Start Apache and MySQL services

Step 2: Create Project Directory

Navigate to your web server's root directory:

    XAMPP: C:\xampp\htdocs\

    WAMP: C:\wamp\www\

    MAMP: /Applications/MAMP/htdocs/

Create a new folder named karu_attendance
Step 3: Copy All Files

Copy all provided PHP, HTML, CSS, and JS files into the karu_attendance folder maintaining the folder structure:
text

karu_attendance/
├── css/
│   └── style.css
├── js/
│   └── validation.js
└── [all PHP and HTML files]

Step 4: Create Database

    Open phpMyAdmin by navigating to http://localhost/phpmyadmin

    Click "New" to create a database

    Database name: karu_attendance

    Choose "utf8mb4_general_ci" as collation

    Click "Create"

Step 5: Create Tables

Run the following SQL queries in phpMyAdmin SQL tab:

Students Table:
sql

CREATE TABLE students (
    id INT PRIMARY KEY AUTO_INCREMENT,
    reg_number VARCHAR(50) NOT NULL UNIQUE,
    fullname VARCHAR(100) NOT NULL,
    email VARCHAR(100) NOT NULL UNIQUE,
    phone VARCHAR(15) NOT NULL,
    course VARCHAR(100) NOT NULL,
    password VARCHAR(255) NOT NULL,
    role ENUM('student', 'lecturer') NOT NULL DEFAULT 'student',
    account_status ENUM('pending', 'verified', 'rejected') NOT NULL DEFAULT 'pending',
    must_change_password TINYINT(1) NOT NULL DEFAULT 1,
    created_by VARCHAR(100) DEFAULT NULL,
    reg_date TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP
);

Attendance Table:
sql

CREATE TABLE attendance (
    id INT PRIMARY KEY AUTO_INCREMENT,
    student_id INT NOT NULL,
    class_date DATE NOT NULL,
    status ENUM('Present', 'Absent') NOT NULL DEFAULT 'Absent',
    FOREIGN KEY (student_id) REFERENCES students(id) ON DELETE CASCADE
);

Step 6: Configure Database Connection

Open config.php and verify/update database credentials:
php

$servername = "localhost";
$username = "root";
$password = "root";
$dbname = "karu_attendance";

Note: For XAMPP default installation, password is often empty string "" instead of "root". Adjust according to your setup.

Step 8: Test the System

    Access the homepage: http://localhost/karu_attendance/index.html

    Test student registration

    Login as lecturer using Staff ID: LEC001 and password: LecSecure@2024

    Approve the test student registration

    Login as student to test attendance signing

Step 9: Verify File Permissions

Ensure the following directories have proper read/write permissions:

    No files require write permissions (no file uploads)

    CSS and JS folders need read access

Attendance button shows already signed	Duplicate submission	Student can only sign once per day

JavaScript validation not working	validation.js path incorrect	Ensure file is in js/ folder

Appendix A: Registration Number Format

The system expects registration numbers to follow this pattern:

Pattern: Letter + 3 digits + / + 4 digits + optional letter + / + 2 digits

Valid Examples:

    S123/0001X/24

    Y987/6543P/21

    C001/2345/23

Invalid Examples:

    123/4567/24 (missing leading letter)

    S12/4567/24 (only 2 digits after letter)

    S123/456/24 (only 3 digits before optional letter)

    S123/4567/244 (3 digits at end instead of 2)

Regular Expression Used:
regex

/^[a-zA-Z]\d{3}\/\d{4}[a-zA-Z]?\/\d{2}$/

Appendix B: Phone Number Format

Kenyan phone numbers must follow this format:

Pattern: Start with 0, then either 1 or 7, then exactly 8 digits

Valid Examples:

    0712345678

    0723456789

    0112345678

Invalid Examples:

    712345678 (missing leading 0)

    071234567 (only 9 digits)

    0812345678 (starts with 08, not 01 or 07)

Regular Expression Used:
regex

/^0[17]\d{8}$/

Appendix C: Password Requirements

Student passwords during registration must meet:
Requirement	Description
Minimum Length	At least 8 characters
Uppercase	At least one uppercase letter (A-Z)
Lowercase	At least one lowercase letter (a-z)
Digit	At least one number (0-9)
Special Character	At least one non-alphanumeric character (!@#$%^&* etc.)

Strong Password Examples:

    KaruStudent@2024

    Pass#123Word

    Secure!24Karu

Weak Password Examples (will be rejected):

    password123 (no uppercase or special character)

    PASSWORD123 (no lowercase or special character)

    Pass123 (too short, no special character)

Regular Expression Used:
regex

/^(?=.*[a-z])(?=.*[A-Z])(?=.*\d)(?=.*[^a-zA-Z0-9]).{8,}$/

View Attendance Statistics
sql

SELECT 
    DATE(class_date) as date,
    COUNT(*) as total_signed,
    COUNT(CASE WHEN status = 'Present' THEN 1 END) as present_count
FROM attendance 
GROUP BY DATE(class_date)
ORDER BY date DESC;
